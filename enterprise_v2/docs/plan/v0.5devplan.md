

# v0.5 开发计划：Skill 自我进化架构精简与落地

## 目标

v0.5 的目标不是继续横向加模块，而是把当前 Skill 自我进化链路压缩成一个更稳定、更可验证、更适合 edge case 学习的闭环。

这一版的核心原则只有三条：

1. 以 `case` 作为唯一学习单元，而不是直接拿 transcript 驱动 skill factory。
2. 以 `patch existing skill` 作为默认演化动作，只有在确实出现新能力缺口时才允许 `create new skill`。
3. 以 `lab` 作为唯一晋升门，任何 skill 变更都必须先经过可验证的包渲染与回归验证。

### 1.2 架构调整原则

相较于现有版本，v0.5 只做三件本质性的收缩：

- 在线程分析与 skill factory 之间插入 `case.yaml` 这一层。
- skill factory 不再直接产最终 skill package，只产 `candidate.yaml + minimal diff`。
- 所有 skill 变更统一进入 `renderer -> lab -> promotion` 链路，不允许绕过验证直接入库。

## 三、v0.5 核心对象定义

### 3.1 新增 `case.yaml`

`case` 是 v0.5 唯一新增的核心对象。

它不是给用户看的产物，而是给 learning loop 使用的标准学习单元。

作用：

- 把 transcript / failure / workflow 去噪后结构化。
- 让 thread analysis 和 skill factory 解耦。
- 让后续统计能够回答“哪个 edge case 被哪个 patch 吃掉了”。

建议最小 schema：

```yaml
case_id: case_20260408_xxx
source:
  task_id: task_xxx
  session_id: sess_xxx
  skill_name: github-pr-workflow
  action_name: create_pr

pattern:
  problem_type: boundary_miss
  summary: "repo 非标准默认分支时原 skill 无法正确定位基线"

boundary:
  trigger_context:
    - "repo has trunk branch"
    - "default branch is not main/master"
  anti_trigger_context:
    - "standard github flow repo"
    - "single branch toy repo"

recovery:
  worked: true
  recovery_path:
    - "detect remote HEAD"
    - "fallback to symbolic-ref origin/HEAD"
    - "recompute diff base"

delta:
  reusable: true
  recommended_evolution: patch
  target_skill: github-pr-workflow
  target_layer: scripts

evidence:
  transcript_refs:
    - "turn_18"
    - "turn_19"
  feedback_ref: "feedback_xxx"
  artifact_refs:
    - "logs/git_error.txt"
```

### 3.2 收瘦后的 `candidate.yaml`

candidate 在 v0.5 不再代表完整 skill 包，而是“最小变更提案”。

```yaml
candidate_id: cand_20260408_xxx
case_id: case_20260408_xxx

decision:
  mode: patch
  reason: "已有 skill 主能力正确，仅缺少默认分支 fallback"

target:
  skill_name: github-pr-workflow
  patch_targets:
    - "scripts/detect_base_branch.py"
    - "SKILL.md"
    - "evals/nonstandard_default_branch.yaml"

changes:
  contract:
    - "add fallback for remote HEAD resolution"
  docs:
    - "update pitfalls and verification section"
  evals:
    - "add regression case for trunk/default mismatch"

risk:
  overreach: low
  portability: medium
  ambiguity: low
```

### 3.3 Skill Package 三层结构

v0.5 明确 skill 变强不是一味加长 `SKILL.md`，而是分层增强：

```text
                Skill Package
┌────────────────────────────────────┐
│  1. SKILL.md                       │
│  什么时候触发、步骤、边界、验证      │
├────────────────────────────────────┤
│  2. actions.yaml                   │
│  输入输出 contract、参数、约束       │
├────────────────────────────────────┤
│  3. scripts/ + evals/              │
│  确定性逻辑 + 回归样例               │
└────────────────────────────────────┘
```

不同 edge case 优先落到不同层：

| edge case 类型 | 优先修改层 |
|---|---|
| 触发错了 / 不该触发却触发 | SKILL.md trigger / anti-trigger |
| 参数结构不稳定 | actions.yaml |
| 有明确 fallback 逻辑 | scripts/ |
| 容易复发，需要长期守住 | evals/ |
| 只是说明不清 | SKILL.md pitfalls / verification |

## 四、主流程具象化

### 4.1 Edge case 进入后的运行流程

```text
[1] 任务执行失败 / 命中边界 case
        │
        v
[2] Runtime 输出 RunFeedbackEnvelope
        │
        v
[3] Case Analyzer 提炼:
    - problem pattern
    - failure boundary
    - successful recovery path
    - reusable delta
        │
        v
[4] 写入 case.yaml
        │
        v
[5] Evolution Decider 判定:
    - ignore
    - patch existing skill
    - create new skill
        │
        ├── ignore → 结束
        │
        ├── patch existing skill
        │       │
        │       v
        │   生成最小 diff
        │
        └── create new skill
                │
                v
            生成 candidate.yaml
                │
                v
[6] Package Renderer
    candidate.yaml → SKILL.md / actions.yaml / evals / scripts
                │
                v
[7] Skill Lab 跑验证
    - trigger 命中是否正确
    - anti-trigger 是否挡住误触发
    - action contract 是否收紧
    - 是否真的修复 edge case
                │
        ┌───────┴────────┐
        v                v
     fail               pass
        │                │
        v                v
  回写 case 状态       Promotion Gate
  记录失败原因          │
                        v
                   Registry Publish
                        │
                        v
                   下次运行直接复用
```

### 4.2 Thread Analysis 与 Skill Factory 的解耦方式

```text
Thread Analysis
    │
    │ 只负责提炼
    v
case.yaml
    │
    │ 只负责学习输入
    v
Skill Factory
    │
    │ 输出最小 candidate
    v
candidate.yaml
    │
    │ 渲染与验证
    v
Skill Package + Lab Result
    │
    ├──────────────┐
    │              │
    v              v
回写 case 状态   更新 skill 能力画像
```

关键原则：

- thread analysis 不直接碰 skill package。
- skill factory 不直接理解完整 transcript。
- 二者通过 `case` 解耦，通过 `outcome` 反哺。

- `CaseRecord`

| `PromotionSubmission` | lab 通过后提交晋升的信息 |

## 六、v0.5 迭代规划

## Phase 1：插入 Case 层

### 目标

把现有 thread/transcript analysis 的输出，从“直接喂 skill factory”改成“先写入 `case.yaml`”。

### 交付项

- 新增 `CaseRecord` schema
- 新增 `state/cases/` 持久化目录
- 实现 `Case Analyzer`：
  - 从 transcript / failure / workflow 中抽取 `pattern / boundary / recovery / delta`
- 所有 learning 链路改为消费 `case.yaml`

### 验收标准

- 任一 edge case 进入后，系统必须先落一份 `case.yaml`
- skill factory 不再直接读取 transcript 原文

---

## Phase 2：收瘦 Skill Factory

### 目标

把现有 factory 从“直接生成 skill package”改成“只生成 candidate proposal”。

### 交付项

- 新增 `CandidateProposal` schema
- 新增 `Evolution Decider`
- 新增判定输出：
  - `ignore`
  - `patch existing skill`
  - `create new skill`
- `Skill Synthesizer` 只输出：
  - `candidate.yaml`
  - `target skill`
  - `minimal diff intent`

### 验收标准

- factory 产物不再直接写完整 skill 目录
- 至少 80% 的历史 edge case 自动落到 `patch` 分支

---

## Phase 3：引入 Package Renderer

### 目标

把 candidate proposal 渲染为标准 skill package。

### 交付项

- 新增 `Package Renderer`
- 支持渲染：
  - `SKILL.md`
  - `actions.yaml`
  - `scripts/`
  - `evals/`
- 明确 patch 模式下的差异渲染策略

### 验收标准

- 任一 candidate 都能被 renderer 渲染为统一 skill 包结构
- patch 模式不会重写无关文件

---

## Phase 4：Lab Only Promotion

### 目标

强制所有 skill 变更必须经过 lab 验证才能晋升。

### 交付项

- lab 增加四类测试：
  - trigger correctness
  - anti-trigger correctness
  - action contract check
  - edge case regression
- 新增 outcome 回写：
  - `ignored`
  - `failed_in_lab`
  - `promoted_as_patch`
  - `promoted_as_new_skill`

### 验收标准

- registry publish 只能消费 lab pass 的产物
- 所有失败 candidate 都必须回写失败原因到 outcome

---

## Phase 5：Patch-first 策略固化

### 目标

防止 skill inflation，建立健康的 skill 进化比例。

### 交付项

- 在 decider 中固化默认策略：
  - patch：80%
  - ignore：15%
  - create：5%
- 统计每周 case 的判定分布
- 给新建 skill 增加强门槛：
  - 必须证明不是已有 skill 的 boundary bug
  - 必须通过 utility / portability 检查

### 验收标准

- 新建 skill 数量明显下降
- 已有 skill patch 成为主流演化路径

---

## 七、v0.5 成功标准

这一版是否成功，不看模块数量，看以下五个指标：

1. 任一 edge case 是否都能沉淀成 `case.yaml`
2. factory 是否已经从“生成器”收缩成“判定器 + 提案器”
3. patch 是否成为默认演化主路径
4. 所有变更是否都经过 `renderer -> lab -> promotion`
5. skill package 是否越来越像“可执行模块”，而不是“越来越长的说明文档”

## 八、最终预期运行状态

```text
白天：
用户任务执行
   ↓
出现 edge case
   ↓
自动生成 case.yaml
   ↓
默认尝试 patch existing skill
   ↓
进入 lab 回归
   ↓
通过后晋升到 registry

晚上：
批量扫描当天 case
   ↓
聚合同类失败模式
   ↓
识别哪些只是 boundary bug，哪些是真正缺能力
   ↓
谨慎 create 新 skill
```

v0.5 的目标不是让系统“长更多东西”，而是让系统“更会把失败压缩成 case，把 case 压缩成 patch，把 patch 变成带 contract 和 eval 的可执行模块”。

这也是这一版最重要的架构判断：

**先把 case 层立起来，再把 factory 收瘦，最后把 promotion 收严。**
