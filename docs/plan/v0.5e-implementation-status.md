## 1. 当前收敛结果

### Stage

截至 2026-04-08，v0.5 计划内的在线主链路、混合检索收口、runtime bounded planner、backend promotion 闭环与 platform 门面兼容层已经全部收口完成。

### 目标

打通并验证下面三条链路：

1. `find_skill -> execute_skill -> runtime -> feedback`
2. `skill.type -> runtime dispatcher -> executor/profile -> feedback metadata`
3. `RunFeedbackEnvelope -> CaseRecord -> CandidateProposal -> ProposalAdapter -> skill.candidate.v1`
4. `feedback -> case -> decision -> proposal/candidate -> promotion intake -> outcome`

### 已完成

- `Skill Engine`
  - 已有 `SkillProjection / FindSkillRequest / ExecuteSkillRequest / RuntimeExecutionResponse`
  - 已有 `EngineSearchService / EngineService`
  - 已有 `find_skill / execute_skill` API 和 platform 暴露
  - registry 已支持 projection 读取和 projection 列表
  - `EngineSearchService` 已收口为 projection-backed hybrid ranking：`base_score / exact_bonus / tree_score / vector_score`
  - 排序 tie-break 已固定为 `is_official desc -> updated_at desc -> skill_id asc`
  - tree/vector enhancer 只影响已通过 projection/filter 的候选技能，不会引入 projection 外结果
  - enhancer 缺失时会自动降级到 registry projection + lexical/exact 排序
  - `find_skill` 最终仍只返回 registry projection，对外 contract 不暴露 manager 原始结构
  - `SkillProjection.from_dict` 已过滤 `package_root / install_root / mounted_path`
- `Skill Runtime`
  - `skill.type` 已进入 install / resolve / feedback
  - runtime 已按 `skill.type` 分发
  - `script` 走标准 contract 路径
  - `agent / ai_decision` 共享受限 `AgentExecutor`
  - agent tool surface 只保留 `read_file / run_action`
  - 多 action skill 已收口到 bounded planner 路径，单 action skill 继续走 fast path
  - `ResolvedAction` 已带 `action_catalog`，planner 只能从声明式 action catalog 中选下一步
  - `ai_decision` 已走与 `agent` 一致的 bounded executor 路径，并稳定回填 `executor / profile / tool_surface / allowed_tools`
  - planner metadata 已透传到 feedback：`planner_used / planner_steps / planner_trace / profile / tool_surface / allowed_tools`
  - `src/orchestrator/runtime` 与 `runtime/src/python/orchestrator/runtime` 已同步 dispatcher/profile/executor/install/resolve/envelope`
- `Backend Enhancement`
  - 已有 `CaseRecord / EvolutionDecision / CandidateProposal / OutcomeRecord`
  - 已有 `LabPromotionResult`
  - 已有 `CaseStore / OutcomeStore / CaseAnalyzer / Decider / ProposalAdapter`
  - 已支持 `RunFeedbackEnvelope -> CaseRecord`
  - 已支持 `CandidateProposal -> skill.candidate.v1`
  - `PromotionSubmission` 已支持 `lineage`
  - 已新增 `LabPromotionOrchestrator`
  - 已支持 `feedback -> case -> decision -> proposal/candidate -> promotion intake -> outcome` 可审计闭环
  - registry 依赖链不可用时，backend enhancement 仍可走审计式本地 promotion intake fallback，不会中断闭环
  - registry `promotion_requests` 已写入 `case_id / proposal_id / decision_mode / lineage_json`
- `Integration Sync`
  - integration 包已同步 engine / lab / registry 对外出口
  - integration runtime 已接回 dispatcher
  - registry app 已同时暴露 projection 路由和 engine 路由
  - source / integration 的 `AgentSkillPlatform` 已同步 factory 相关旧门面
  - `detect_platform_paths()` 已恢复 contracts/runtime/lab/factory/registry 兼容路径字段

### 本轮收口结论

- `v0.5-skill-engine-reframe.md`
  - A-E 阶段目标已经全部落到当前仓代码，并由测试固定下来。
- `v0.5a-skill-engine-delivery-plan.md`
  - projection schema、registry projection、统一 find/execute service、platform 暴露已经完成。
- `v0.5b-skill-runtime-delivery-plan.md`
  - `skill.type` 贯穿 install/resolve/dispatch/feedback、script runtime closure、bounded planner、`ai_decision` profile 已完成。
- `v0.5c-backend-enhancement-delivery-plan.md`
  - case、proposal、adapter、outcome、promotion integration 已完成并串成可审计闭环。

### 后续边界

- v0.5 已经完成，但这不等于“生产化工作全部完成”。
- 若继续进入 v1 / production hardening，后续重点应放在：
  - 更强的远端/大规模搜索基础设施，而不是改动当前 projection contract
  - 更稳健的 planner 依赖治理与可观测性，而不是放宽工具面
  - 更完整的 promotion review / approval 产品化界面，而不是绕过 `ProposalAdapter`

### 是否允许进入下一阶段

允许，并且当前已经具备进入下一阶段的代码与文档基线。

理由：

- 在线主链路已经跑通并通过目标测试集
- hybrid search、bounded planner、promotion 闭环都已经落到当前仓实现
- source / runtime mirror / integration 三套实现已经同步验收

## 3. enterprise_v2 基础设施连调状态

截至 2026-04-09，`enterprise_v2/` 已完成一轮真实 Docker 基础设施连调验证。

### 已验证链路

- `docker compose up --build -d`
- `GET /healthz`
- `GET /readyz`
- `POST /publish`
- `GET /skills`
- `GET /skills/projections`
- `GET /internal/jobs/{job_id}`
- `POST /execute-skill`
- `POST /feedback`

### 真实连调过程中暴露并已修正的问题

- 运行期容器继承宿主机代理变量，导致 MinIO 请求被错误转发到外部代理，已在 compose 里显式清空 `HTTP_PROXY / HTTPS_PROXY`
- API 启动时被旧 lab 包入口的重型导入拖住，已把 v2 的 `lab/__init__` 精简为最小入口
- `publish` 在 Postgres 下会因父记录未先 flush 导致外键失败，已修正 repository 写入顺序
- manifest 中的日期字段直接写 JSON 列会失败，已改为 JSON-safe dump
- readiness 之前直接抛 500，现已收口为可读的失败返回和 503 状态

### 当前仍需留意的事项

- `skill_versions` 当前仍使用裸 `version_id` 作为主键
- 对于长期多人使用或多团队共库场景，建议升级为更稳妥的版本键模型
- `publish.source` 仍然依赖 API 服务可见路径，尚未做成浏览器上传或对象存储直传模式
